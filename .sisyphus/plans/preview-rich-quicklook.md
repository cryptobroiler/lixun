# Preview rich QuickLook — implementation plan

GTK4 + xdg-toplevel + custom PDF viewer. Pragmatic path, no Qt6, no cxx-qt
risk. Target: QuickLook-level interaction (pan / zoom / paginate / text
selection / image-region selection) for all currently-supported formats,
with a clear external-app fallback for niche formats (DjVu / Jupyter /
SQLite) deferred to v2.

## Why this plan, not Qt6/QML

Verified May 2026:

- `cxx-qt` (KDAB) is pre-1.0, KDAB themselves do not ship it for client
  work, qmetaobject's author moved to Slint. Zero production Rust+Qt6
  desktop apps in the wild.
- `cxx-qt-lib` covers QtCore/QtGui only. **QtPdf has no Rust bindings**
  — would have to manually wrap ~20 C++ classes, defeating the entire
  "QtPdf out of the box" argument.
- LayerShellQt and `xdg-foreign-v2` have no Rust integration on the Qt
  side either.
- `libreoffice-pure` (the "100× faster" crate from the 2026-05 research)
  is 1 month old, 3 GitHub stars, 13 test fixtures, PPTX charts at 42 %
  fidelity. Not production-ready.

The Oracle 6–9-week Qt6 estimate assumed those bindings existed. They
don't. Realistic Qt6 effort is 12–18 weeks plus first-mover risk.

Equivalent custom GTK4 PDF viewer effort, with no FFI risk and a
production layer-shell-equivalent (`xdg-toplevel` + `xdg-foreign-v2`),
is 6–8 weeks. We take that path.

## Scope

In scope (Phase 1 → 4):

- Migrate preview window from `wlr-layer-shell` to `xdg-toplevel` +
  `xdg-foreign-v2` transient parenting. Fixes z-order, fcitx5, fullscreen,
  multi-monitor edge cases without changing the toolkit.
- Build a custom GTK4 PDF viewer on `poppler-rs`, with text selection,
  search, zoom/pan via gestures, and pagination. Reuse Loupe's gesture
  pattern verbatim where possible.
- Upgrade the office plugin to render to PDF (LibreOffice headless,
  cached) and route through the new PDF viewer. All pages, full
  interaction.
- Upgrade text / code / image / markdown plugins to share the same
  gesture and selection vocabulary.
- Add `Action::OpenUri` external-app fallback for DjVu / EPUB / Jupyter
  / SQLite (host-side, plugin-agnostic per AGENTS.md).

Out of scope (deferred):

- DjVu / EPUB / Jupyter / SQLite native viewers. External app via
  `xdg-open` for v1; revisit when there is real demand.
- Qt6 / Slint migration. Reconsidered if cxx-qt 1.0 lands with QtPdf
  bindings AND a production user appears.
- Preview plugin sandbox / untrusted-tier isolation. Tracked separately
  in roadmap §5.

## Modularity invariants (AGENTS.md)

Every change below must keep these true:

- Host (`lixun-preview-bin`, `lixun-daemon`, `lixun-gui`) never names a
  concrete plugin id, file extension, or category.
- The trait crate (`lixun-preview`) gains methods with sensible defaults
  when host needs to ask plugins something new. No host-side switches
  on plugin id.
- Layer-shell-vs-`xdg-toplevel` is a host-policy decision. Plugins do
  not know which window kind hosts them.
- External-app fallback is plugin-agnostic. Plugins emit
  `Action::OpenUri { uri: "file://..." }` and the host dispatches via
  `xdg-open`. Plugins do not name applications.
- New PDF viewer widget lives inside `lixun-preview-pdf` (and any
  shared widget code lives in a plugin-agnostic helper crate).
- Office plugin owns the docx/xlsx/pptx → PDF conversion. Host does
  not learn what `soffice` is.

Invariant test before merging any phase:

```
git grep -nE 'thunderbird|libreoffice|soffice|evince|okular|mpv|xdg-mime' \
  -- crates/lixun-preview-bin crates/lixun-daemon crates/lixun-gui \
     crates/lixun-preview/src
```

Must return zero executable matches outside docstrings / comments.

## Phase 1 — xdg-toplevel migration (1 week)

Goal: drop `wlr-layer-shell` from the preview window, switch to
`xdg-toplevel` + `xdg-foreign-v2` transient parenting. Launcher stays
on layer-shell. This fixes:

- z-order chaos (preview ending up under launcher in 6a/6c).
- fcitx5 candidate popup z-order under launcher.
- Fullscreen edge cases (layer-shell overlay + fullscreen apps clash).
- Multi-monitor positioning quirks under KWin.

It does not deliver new viewer features; it is the structural
prerequisite that lets us build them on a stable surface.

### 1.1 Preview window kind

`crates/lixun-preview-bin/src/main.rs` lines 536–556 currently call:

- `init_layer_shell()`
- `set_layer(Layer::Overlay)`
- `set_anchor(Edge::*, false)` × 4
- `set_keyboard_mode(KeyboardMode::None)`

Replace with `xdg-toplevel`:

- Remove `gtk4-layer-shell` dependency from `lixun-preview-bin/Cargo.toml`.
- Window is a plain `gtk::ApplicationWindow`, `set_decorated(false)`,
  `set_resizable(false)` for v1, `set_default_size(W, H)` from the
  plugin's `SizingPreference`.
- Skip taskbar via the existing GTK `app-id` matching the desktop file's
  `NoDisplay=true` shadow (or via `gtk_window_set_skip_taskbar_hint` on
  X11 fallback; on Wayland the `app_id` is enough plus a `.desktop`
  with `NoDisplay=true`).
- Window stays positioned by the compositor; we do not anchor.

### 1.2 Transient parenting via xdg-foreign-v2

The launcher exports a handle, the preview imports it as parent. This
keeps the preview above the launcher without needing layer-shell
ordering, and it follows the launcher across virtual desktops / monitors.

Crates:

- Use `wayland-protocols` crate for `xdg-foreign-v2` client side. The
  protocol is supported on KWin / Hyprland / sway / Niri / labwc /
  Mutter (yes, including GNOME — the launcher itself is the part that
  GNOME cannot host, not the preview).
- `gtk4-rs` exposes `gdk::Wayland*` types we need to fish out the raw
  `wl_surface`.

Flow:

1. Launcher process: on first preview spawn, call
   `zxdg_exporter_v2.export_toplevel(launcher_surface)` and receive a
   handle string (UUID-like).
2. Daemon passes the handle string to preview-bin via a new
   `PreviewCommand::SetParent { handle: String }` IPC message (added
   to `lixun-ipc::PreviewCommand`).
3. Preview-bin: on `SetParent`, call
   `zxdg_importer_v2.import_toplevel(handle)` and then
   `set_parent_of(preview_surface)` on the resulting `xdg_imported`.
4. From then on the compositor stacks the preview above the launcher
   automatically and follows it.

Fallback when `xdg-foreign-v2` is unavailable (rare in 2026, possible
on niche compositors): preview opens as a plain centred toplevel. We
log a warning once per session, do not block.

### 1.3 IPC additions (`lixun-ipc/src/preview.rs`)

Add to `PreviewCommand`:

```
SetParent { handle: String },     // xdg-foreign-v2 handle from launcher
ClearParent,                       // launcher tells preview the handle is gone
```

Add to `PreviewEvent`:

```
ParentLost,                        // compositor revoked the imported handle
```

No new daemon state machine work; `SetParent` rides on the existing
`PreviewLifecycle::{Starting, Ready}` flow. The daemon buffers `SetParent`
until `Ready` arrives, same as it already buffers `ShowOrUpdate`.

### 1.4 Launcher-side changes (`lixun-gui`)

- Drop the `Layer::Top` promotion from PR1 6a — that was a workaround
  for layer-shell same-layer ordering. Launcher returns to
  `Layer::Overlay` (or stays at `Top`; verify on KWin/Hyprland it
  doesn't regress fcitx5 — it shouldn't, because preview is no longer
  on the layer-shell hierarchy at all).
- On preview spawn, export the launcher surface via xdg-foreign-v2 and
  send the handle through the daemon to preview-bin.
- On launcher close, send `ClearParent` through the daemon (best-effort;
  preview will also notice via `xdg_imported.destroy`).

### 1.5 Keyboard / focus model

- Launcher: keep `KeyboardMode::OnDemand` — unchanged.
- Preview: with no layer-shell, there is no `set_keyboard_mode` call.
  The window is a regular `xdg-toplevel`. We call
  `set_can_focus(false)` on the root and rely on launcher staying
  focused. Escape / Space / typing-while-preview-open behaviour from
  PR1 keymap.rs stays as-is — those handlers are launcher-side and
  don't care which window kind the preview uses.
- Focus-loss-close stays REMOVED (PR1 §1.1 fix #6).

## Phase 2 — custom GTK4 PDF viewer (4 weeks)

Goal: replace the current `lixun-preview-pdf` (Cairo direct draw, no
selection, no search, basic zoom) with a QuickLook-class viewer:

- Pinch-to-zoom and Ctrl+scroll zoom.
- Drag-to-pan.
- Page navigation: continuous scroll with snap-to-page, plus
  PageUp/PageDown / arrows / number-jump.
- Text selection with mouse drag, Cmd/Ctrl+C copy.
- Text search with highlight + next/prev.
- Outline / bookmarks sidebar — **DEFERRED** (no safe poppler-rs

  shown when document has a non-trivial outline.

Reference: GNOME Loupe's gesture pattern (verified production), GNOME
Papers' Poppler integration (text selection model, search hits),
combined into one widget local to `lixun-preview-pdf`.

### 2.1 New widget structure

A new internal widget tree, lives entirely inside `lixun-preview-pdf`:

```
PdfView (GtkBox, vertical)
├── PdfToolbar (GtkActionBar)        // page input, search, zoom controls
└── PdfScroll (GtkScrolledWindow)
    └── PdfCanvas (GtkDrawingArea)   // custom snapshot of current pages
```

Why one canvas instead of one `GtkPicture` per page:

- Selection rectangles cross page boundaries; one snapshot is simpler.
- Re-render on zoom is one invalidation, not N.
- Loupe demonstrated this is fast enough on GTK4's snapshot model.

Trade-off: continuous scroll requires us to track which pages are in
the viewport and render only those. Worker pool handles that.

### 2.2 Render pipeline

Worker model (off the GTK thread):

```
main thread          worker thread             page cache
─────────             ─────────────             ──────────
viewport changes  →   request {page, zoom, dpi} →  HashMap<(page,zoom_bucket), Arc<gdk::Texture>>
                  ←   gdk::Texture (sent via    ←  cached if hit
                      async_channel)               rendered if miss
queue_draw()
```

- `poppler-rs` `Page::render_for_printing` into a `cairo::ImageSurface`,
  then convert to `gdk::MemoryTexture` for snapshot.
- Worker pool: 2 threads (configurable). One thread = visible page,
  second = adjacent prefetch.
- Cache key: `(page_index, zoom_bucket)` where `zoom_bucket` is the
  zoom rounded to 25 % steps so we don't re-render on every pixel of
  zoom drag. Visible at the final zoom-stable level we render once
  more at the exact zoom for crispness.
- LRU eviction at ~64 MB texture cache (configurable).
- Cairo ABI workaround stays — `poppler-rs` 0.26 links cairo 0.22,
  gtk4-rs links cairo 0.20; we already isolate the boundary.

### 2.3 Gestures (Loupe pattern, verified)

```
let zoom = gtk::GestureZoom::new();
zoom.connect_scale_changed(|gesture, scale| {
    // accumulate, clamp [0.25, 16.0], rebucket
});
let drag = gtk::GestureDrag::builder().button(0).build();
drag.connect_drag_update(|_, dx, dy| {
    // pan: adjust GtkAdjustment of the ScrolledWindow
});
let scroll = gtk::EventControllerScroll::new(
    gtk::EventControllerScrollFlags::VERTICAL
);
scroll.connect_scroll(|_, _dx, dy| {
    // Ctrl held → zoom toward cursor; else → scroll
});
```

Gesture-touchpad differentiation taken straight from Loupe:
`gesture.device().map(|d| d.source())` switches between
`InputSource::Touchpad`, `Touchscreen`, `Mouse`. For touchscreens we
allow pinch + pan simultaneously; for touchpads we use Ctrl+scroll for
zoom. Mouse: wheel scrolls, Ctrl+wheel zooms.

### 2.4 Text selection

Poppler gives us per-page text + per-glyph rectangles via
`Page::get_text_layout`. We:

1. On `mouse-down` inside canvas: hit-test which page + glyph index
   the pointer is over. Store as `selection_anchor`.
2. On `mouse-drag`: hit-test current pointer, build a
   `SelectionRange { from: (page,glyph), to: (page,glyph) }`.
3. On render: after page raster, paint translucent blue rectangles
   over selected glyphs (snapshot overlay, not into the texture).
4. On Ctrl+C: collect glyphs in range across pages, join with newlines
   between pages, push to `gdk::Display::default().clipboard()`.

Cross-page selection is the only tricky part. Resolved by storing the
selection in (page, glyph) space, not pixel space; the renderer maps
to pixels per page.

### 2.5 Text search

`PdfDocument::search_text(query)` is not in `poppler-rs`. We do it
manually:

- For each page, `page.text()` gives the full text.
- Search via `aho-corasick` for case-insensitive substring; collect
  match offsets.
- Map offsets back to glyph rectangles via `page.get_text_layout`.
- Paint matches with a different highlight color (yellow); the current
  match gets an orange outline.
- Toolbar: search field, match counter (`3 / 17`), prev/next buttons.
- Enter in search field → next match. Shift+Enter → prev. Esc closes.

### 2.6 Page navigation

- Continuous scroll with snap: scroll position in document space,
  current page = page whose centre is closest to viewport centre.
- Toolbar shows `7 / 23`, editable. Number entry → jump.
- PageUp/Down: ±1 page, snap to top of next page.
- Home/End: page 1 / last page.

### 2.10 Oracle revision (post-Momus, pre-implementation)

After Phase 1 landed and PR2a entered scope-freeze, an architecture
consultation surfaced seven concrete corrections to §2.1–§2.9. The
original plan stays Momus-approved; this subsection is additive and
overrides the originals only where explicitly noted.

**Q1 — canvas API.** `gtk::DrawingArea::set_draw_func` hands you a
GTK cairo-0.20 `Context`, which reintroduces the cairo-22-vs-cairo-20
ABI split the existing dumb plugin already isolates. PR2a uses a
**custom `gtk::Widget` subclass** for `PdfCanvas` with `WidgetImpl::
snapshot()` and `snapshot.append_texture(...)` instead. The widget's
allocation equals full document size in document coordinates;
`ScrolledWindow` clips. No cairo `Context` ever crosses the GTK
boundary.

**Q2 — render thread model.** `poppler::Document` is `!Send`
(GObject refcounted). PR2a uses **per-owner `Document`**: the main
thread holds one for text/layout queries (outline deferred), plus two long-
lived render workers each holding their own `Document` for the same
file. Worker A renders the visible page, worker B prefetches
adjacent. On Hit change the host bumps an epoch, drops pending
jobs, and all three owners reopen. Do **not** open `Document` per
render request, do **not** funnel everything through one serial
worker.

**Q3 — gesture conflict.** Risk is medium only if `ScrolledWindow`
kinetic scrolling is enabled. Mitigation: call
`scroll.cancel_deceleration()` on `GestureZoom::begin` and
`GestureDrag::drag_begin`, **or** simply
`scroll.set_kinetic_scrolling(false)` on `PdfScroll`. Pick whichever
binding surface is cleaner on gtk4-rs 0.11.

**Q4 — zoom-toward-cursor math.** Compute the anchor in **document
space**, not page-relative ratios:

```
doc_x = (hadj.value() + cursor_x) / old_zoom;
doc_y = (vadj.value() + cursor_y) / old_zoom;
// apply new zoom and relayout canvas
hadj.set_value(doc_x * new_zoom - cursor_x);
vadj.set_value(doc_y * new_zoom - cursor_y);
```

Critical: clamp the new adjustment values **after** recomputing
canvas/document size, not before, or the anchor drifts near
bottom/right edges.

**Q5 — current-page rule.** Nearest-center rule with **tie-break to
previous page**. Add an explicit deadband only if QA shows real
midpoint flicker. Do not preemptively use 33/67 hysteresis — it
makes navigation feel sticky.

**Q6 — selection model.** Build `PageTextCache { text, glyph_rects }`
once per page on document open. `SelectionRange` indices reference
this cache, not render output. Cross-page ordering is
`(page_index, glyph_index)`. Glyph indices stay stable for one
opened `Document` instance, so cache once. Plan §2.4 stays correct;
this subsection just nails down the cache shape.

**Q7 — search performance.** Lazy with background warmup. On
document open, kick off a low-priority pass filling
`PageTextCache.text` for every page. First search hits whatever's
cached immediately; remaining pages stream in and the result count
updates incrementally. ~10 MB for 600 pages is acceptable; blocking
preview-open on 600 `page.text()` calls is not.

**Q8 — encrypted-PDF detection.** Before promising precision in PR2a
copy, write a fixture test against `tests/fixtures/preview/pdf/
sample-encrypted.pdf` to inspect the actual `glib::Error` returned
by `Document::from_file`. If poppler-rs 0.26 exposes a
distinguishable typed code, branch on it and show "Cannot preview
password-protected document". If not, fall back to a generic
"Cannot preview PDF" for all open failures and revisit when the
binding tightens. **No string-matching on error display text.**

**Q9 — `PreviewCapabilities` shape.** Lock the struct now with
forward-compat hooks so Phase 4 (text/code/image/markdown) can
extend without touching every plugin:

```rust
#[derive(Default, Debug, Clone, Copy, PartialEq, Eq)]
#[non_exhaustive]
pub struct PreviewCapabilities {
    pub text_selection: bool,
    pub text_search: bool,
    pub paginated: bool,
    pub zoomable: bool,
}

impl PreviewCapabilities {
    pub fn union(self, other: Self) -> Self { /* OR all fields */ }
}
```

Host reads caps, never branches on plugin id (AGENTS.md invariant).

**Q10 — PR2a effort revised.** Plan §2.9 budgeted PR2a at 1.5 weeks.
**Realistic estimate is 2–2.5 weeks.** Likely overrun source is
**not** gestures — it is the trio of (1) widget subclass with
correct lifecycle, (2) epoch handling and worker job cancellation,
(3) `ScrolledWindow` sizing/clipping interaction with the document-
sized canvas allocation. Lock those three before touching gestures
or text, and the rest stays bounded.

**Additional `update()` requirement (not in original §2).** The
`PreviewPlugin::update(&self, &Hit, &gtk::Widget)` trait method
already exists in `lixun-preview/src/lib.rs` and exists explicitly
to keep arrow-key scrubbing instant by mutating the existing widget
tree instead of rebuilding. The current dumb PDF plugin opts out
via the `UpdateUnsupported` sentinel; that is fine for a one-page
texture render but **regresses QuickLook semantics** for the rich
viewer. PR2a MUST implement `update()` for `PdfPreview`:
`DocumentSession::replace_path(new_path)` bumps epoch, all three
`Document`s reopen, texture cache clears, viewport resets to top.
Without this, every Hit change triggers a full widget rebuild and
arrow-key navigation feels laggy.

## Phase 3 — office plugin (rich PDF via LibreOffice)

Goal: docx / xlsx / pptx / odt / ods / odp / doc / xls / ppt / rtf get
the same rich preview as PDF. Same interaction surface: pan / zoom /
paginate / text selection / search.

Approach: convert to PDF in the office plugin, then reuse the new PDF
viewer widget (Phase 2). The user explicitly required full document
interaction for office docs — PNG-only is rejected.

### 3.1 Conversion pipeline

Stay on `soffice --headless --convert-to pdf`:

- Production-proven, 100 % fidelity, used by GNOME Sushi the same way.
- The "100×-faster pure-Rust libreoffice-pure" alternative is 1 month
  old, single maintainer, 13 test fixtures, PPTX charts at 42 % — not
  ready. Re-evaluate in 6 months.
- For XLSX *data extraction* (not preview rendering) the calamine
  crate is production-ready, but that is a different feature — not in
  this phase.

Office plugin pipeline:

1. `build()` returns the same spinner placeholder as today.
2. Worker thread spawns `soffice --headless --convert-to pdf
   --outdir <cache_dir>/office <path>` with 30 s wall-clock timeout
   (up from 15 s — multi-page PDFs need it).
3. Cache key: `SHA-256(path_bytes || \"\\0\" || mtime_nanos)` hex,
   stored as `<cache_dir>/office/<key>.pdf`. Same scheme as today; the
   PNG cache is invalidated on first run because the extension differs.
4. Cache hit: skip soffice, jump to step 6.
5. Worker pushes the pdf path onto an `async_channel`.
6. `glib::spawn_future_local` task awaits, then swaps placeholder for
   the new `PdfView` widget (Phase 2) loaded with the cached pdf.

The office plugin owns the pipeline. Host knows nothing about
LibreOffice. AGENTS.md modularity invariant holds.

### 3.2 Reuse the PDF widget

The new `PdfView` (Phase 2.1) is the natural reuse target. Move it
out of `lixun-preview-pdf` into a shared crate
`lixun-preview-pdf-view` (or expose it via `lixun-preview-pdf` as a
public widget — same effect, lighter Cargo graph).

Decision: expose it via `lixun-preview-pdf` as `pub use widget::PdfView;`
from the crate root. The office plugin depends on `lixun-preview-pdf`
just for the widget type, not for plugin registration. No new crate.

This keeps the trait abstraction clean: both pdf and office plugins
register independently via `inventory::submit!`, neither one is
"primary".

### 3.3 Per-format quirks

- **Spreadsheets (xlsx / xls / ods)**: `soffice --convert-to pdf`
  paginates by sheet + print-area; this is what Numbers / Sheets /
  Excel preview do too. Acceptable.
- **Presentations (pptx / ppt / odp)**: one slide per page in the
  generated PDF. Acceptable, matches expected slide-deck preview.
- **Legacy formats (doc / xls / ppt)**: soffice handles them; same
  pipeline.
- **Encrypted office docs**: soffice prompts on stdin and hangs. Detect
  by checking the input file with `olefile`-style header sniff (or
  just trust the timeout). Show "Cannot preview password-protected
  document" placeholder.

### 3.4 Cache behaviour

- TTL: indefinite within the cache dir. The mtime in the key handles
  staleness.
- Eviction: piggyback on the existing preview cache cleanup (LRU at
  ~256 MB total preview cache, configurable). Office PDFs are larger
  than the old PNGs (a 50-slide PPTX → ~5–15 MB PDF), so the cap may
  need bumping. Document the new typical sizes in the cleanup logic.
- Pre-warm: not in v1. Considered for v2 once we have indexer hits.

### 3.5 Multi-page is no longer a documented limitation

Today's office plugin docs say "soffice --convert-to png emits only
page 1; documented v1 limitation". After this phase, the limitation is
removed. Update the office plugin's module docstring.

### 3.6 Effort

- Pipeline switch (png → pdf): 1 day.
- Cache scheme bump + cleanup tuning: 1 day.
- `PdfView` reuse + integration test: 1 day.
- Per-format soffice flag tuning (page size for spreadsheets, etc.):
  1 day.
- QA across the 10 supported extensions, encrypted-doc edge case,
  huge-PPTX edge case: 1 day.

Total: 1 week. Single PR: `feat(preview-office): convert to pdf and
reuse the rich pdf viewer`.

## Phase 4 — text / code / image / markdown unification (1 week)

Goal: every plugin that produces user-readable content uses the same
interaction vocabulary. Pinch-zoom, text selection, copy, search where
applicable.

### 4.1 Text plugin

Already uses `gtk::TextView` which has selection + copy out of the box.
Additions:

- Ctrl+F search bar (toggle via Ctrl+F or toolbar). Search via
  `gtk::TextSearchContext`, highlight matches, prev/next.
- Ctrl+= / Ctrl+- / Ctrl+0 to change font size (zoom equivalent).
- Pinch-zoom on touchpads: GestureZoom on the TextView, scaling the
  font size, not the viewport.
- Returns `PreviewCapabilities { text_selection: true, text_search: true,
  zoomable: true, paginated: false }`.

### 4.2 Code plugin

Currently uses syntect-rendered HTML in a WebKit (or a custom widget;
verify). Switch to `sourceview5::View` if not already — production-grade
text editor widget with:

- Selection + copy out of box.
- Search & replace built-in.
- Line numbers.
- Configurable themes (we already read theme from `cfg.section`).
- `gutter`, `bracket-matching`, `highlight-current-line` — all free.

Pinch-zoom = font size, same as text plugin.

Returns `{text_selection: true, text_search: true, zoomable: true}`.

### 4.3 Image plugin

Largest existing GTK widget surface (16 references). Reuse the PdfView
widget's gesture pattern verbatim, but on a `gtk::Picture`:

- Pinch-zoom and Ctrl+scroll zoom, clamped [0.1, 32].
- Drag-to-pan (when zoomed in beyond viewport).
- Image-region selection: shift+drag draws a marquee rectangle.
  Result available via `Action::CopyImageRegion { rect }` (not in v1
  interaction; the rect is just shown for now). Actual region copy
  goes into v2.
- Rotation: R / Shift+R rotate 90° (data only, not on disk).
- Reset: 0 / double-click.

Returns `{zoomable: true, ...}`.

### 4.4 Markdown plugin (new sub-feature)

Markdown is currently rendered by the text plugin as plain text. Add a
small `markdown_render` mode in the text plugin (gated by extension),
using `pulldown-cmark` to parse and a `gtk::TextView` with markup
tags for rendering:

- Headings: bold, larger font.
- Lists: indented bullets.
- Code blocks: monospace + light background.
- Links: blue + underlined; click → `Action::OpenUri { uri }`.
- Inline emphasis: italic / bold.

No table / image rendering in v1 (would need WebKitGTK; out of scope).

The markdown sub-mode is a strategy inside the text plugin, not a new
plugin — keeps the plugin surface small. If future needs grow (full
HTML, MathJax), promote to its own plugin then.

### 4.5 Email plugin

Already shows rich content; mostly leave alone. Verify selection works
across body text. Returns `{text_selection: true}`.

### 4.6 AV plugin

Today has a gtk::Video bug workaround for audio-only files
(`unsafe vbox.set_data::<gtk::MediaFile>(...)`). The xdg-toplevel
window kind makes fullscreen video preview saner; revisit the audio-only
hack to see if it's still needed (likely yes, that bug is upstream).

Capabilities: `{}` (transport controls live in the video widget).

### 4.7 Effort

- Text + search: 1 day.
- Code + sourceview5 wiring: 1 day.
- Image gestures + region marquee (display only): 1 day.
- Markdown sub-mode: 1 day.
- QA across all preview categories on the smoke matrix from §1.6: 1 day.

Total: 1 week. Ship as one PR per plugin (4 PRs), or one bundled PR if
the diff stays small. Prefer per-plugin to keep blast radius bounded.

## Phase 5 — external-app fallback for niche formats (3 days)

Goal: DjVu / EPUB / Jupyter / SQLite get *some* preview path without
us building four custom viewers. The path is "press Enter / Space and
the right native app opens, parented to the launcher".

### 5.1 The mechanism is generic

We already have `Action::OpenFile { path }` and
`Action::OpenUri { uri }`. Both dispatch through the host's
`execute_action` which calls `xdg-open`. Per AGENTS.md we must NOT
name a specific app on the host side; `xdg-open` consults the user's
MIME associations.

`xdg-open path/to/file.djvu` already works on every supported
compositor — that part is fine.

What we add:

- **xdg-foreign-v2 parenting** for the externally-opened window. The
  preview-bin already has the imported parent handle (Phase 1). Pass
  the same handle to the external app via the `XDG_TOPLEVEL_PARENT`
  env var (no such standard yet — set it speculatively; well-behaved
  Qt/GTK apps starting to honour it in 2026). This is best-effort:
  the main UX is the launcher hint "press Enter to open in <app>".
- **A generic preview placeholder** for "format we don't render
  inline yet". Looks like the office spinner but with copy "Press Enter
  to open in your default <DjVu / EPUB / Jupyter / SQLite> app".

The placeholder is rendered by a new tiny plugin
`lixun-preview-external` that:

- Matches any hit with a `match_score()` of `1` (lowest non-zero — so
  any specific plugin wins).
- Reads a config map `extensions_to_label` from `[preview.external]`
  TOML section. Default ships with djvu / epub / ipynb / sqlite / db /
  djv / opf entries.
- `build()` returns the placeholder widget.
- Returns `PreviewCapabilities::default()` (no inline interaction).

The host does NOT learn what an .ipynb is. The plugin's config tells
it which extensions to claim.

### 5.2 Why not build native viewers for these

- **DjVu**: niche, ~2 % of users. Existing apps (DjView4, Okular) are
  excellent. No demand signal yet.
- **EPUB**: covered well by Foliate / Calibre. Users of EPUB workflows
  already have an app installed.
- **Jupyter**: rendering well requires a full HTML stack + code
  execution sandboxing. JupyterLab is the canonical viewer.
- **SQLite**: DB Browser for SQLite is the standard. A scrolling table
  viewer that respects column types is a multi-week project on its
  own.

We revisit any of these if user telemetry (or just user requests)
shows real demand. Not before.

### 5.3 Effort

- New `lixun-preview-external` crate: 0.5 day.
- Placeholder widget + config wiring: 0.5 day.
- xdg-foreign-v2 env-var passthrough (best-effort): 1 day.
- QA on the four formats with their default apps installed: 1 day.

Total: 3 days. Single PR: `feat(preview): external-app fallback plugin
for djvu / epub / ipynb / sqlite`.

|------|-------------|--------|------------|
